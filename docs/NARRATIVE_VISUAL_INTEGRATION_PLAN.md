## 0. 結論

本作には物語が不足しているのではない。すでに5章、初回夢＋夢渡り7篇、郷の声18本、非塔39地域の四段階縁起、3結末がある。弱点は、それらが別々の長文画面や一覧へ蓄積され、**探索・戦闘・帰還・郷の生活中に同じ問いとして響いていないこと**である。

新しく長編を足す前に、既存の強い神話を次の一文へ統合する。

> **継ぐとは、同じ犠牲を繰り返すことか。それとも、ひとりで背負ってきたものを分け合うことか。**

固定神話は「汐里と玄冬の千年の二重奏」。可変の主人公はプレイヤーの燈守家である。各世代の選択、看取り、形見、帰還が固定神話を鏡写しにし、最終3択は突然の倫理問題ではなく、何代もかけて家が考えてきた答えになる。

実装の要点は次の5つ。

1. 夢渡りの順序抜けを防ぎ、弐〜終へ固有画像7点を割り当てる。
2. 長文sceneを増やさず、各画面へ3〜12秒の「物語の残響」を置く。
3. 地域縁起、章、夢、郷の声を一つの`NarrativeStage`で整理し、同じ真相を先に言わない。
4. 内部ID `cut / save / inherit`（表示は`送る / 救う / 継ぐ`）への家の傾きを記録するが、結末はロックしない。最後に家譜の実績を根拠として見せる。
5. 全物語を中断可能・再読可能にし、重要情報を画像、色、音だけへ依存させない。

## 2. 物語方向の比較と採用

| 案 | 構成 | 感情曲線 | 適合理由 | 弱点 |
|---|---|---|---|---|
| A. 千年の二重奏 | 汐里と玄冬を固定主人公にし、真相を中心に進む | 敵意→親近→幸福→不安→破局→介抱→選択 | 既存dream本文と画像の力を最大化 | プレイヤーの一族が観客になりやすい |
| B. 家譜の群像 | 各世代の当主、親子、形見を中心に進む | 誕生→成長→決断→喪失→継承 | procedural familyが本作固有の主人公になる | 固定人物が薄く、長編の推進力が散る |
| C. 常夜の謎解き | 地域痕跡と矛盾を集め、玄冬の正体を推理する | 疑問→仮説→反証→真相→判断 | 探索そのものへ物語を置きやすい | 既存夢で真相を直接語るため、二重説明になりやすい |

### 採用: Aを背骨、Bを鏡、Cを届け方にする

ユーザーから方向選択を求めず自走するため、上記hybridを採用する。固定神話は感情を支え、各世代はその神話と似た選択を異なる規模で繰り返す。探索の痕跡は「答え」ではなく、次の夢を別の意味に見せる問いを渡す。

## 3. 共通の背骨

### 3.1 プレミス

八季で死ぬ燈守家が代を継ぎ、千年前から星喰いを抱き止める家祖・汐里へ辿り着き、同じ犠牲を終わらせるか、分け合うか、引き受け直すかを選ぶ。

### 3.2 テーマ

- 表層: 命は短くても、受け渡されたものは千年へ届く。
- 深層: 愛は相手の苦しみをひとりで背負うことではなく、言葉にして分けること。
- 反テーマ: 「守る者は黙って犠牲になればよい」「怪物は事情を知らず斬ればよい」。

### 3.3 主人公の欲求と欠落

| 主体 | 欲求 | 欠落 | 変化 |
|---|---|---|---|
| プレイヤーの燈守家 | 夜を終え、家族を少しでも長く生かしたい | 八季の呪いを「倒すべき敵のせい」とだけ捉える | 家祖と敵の関係を知り、家として答えを選ぶ |
| 汐里 | 玄冬を鎮め、郷も守りたい | 自責から責任を共有できず、ひとりで千年抱える | 子孫へ選ぶ権利と看取りを委ね、琵琶を置く |
| 玄冬 | 汐里の唄を聴き、地上にいたい | 飢えを理解・説明できず、迷惑をかける前に離れられない | 怪物の面の下に「名をもらった子」が残っていたと明かされる |
| 綴 | 一族の生を一字も漏らさず記したい | 記録できても、汐里の孤独を終わらせられない | 最後の頁を生者へ渡し、記録者から証人になる |
| 郷 | 大燈籠と燈守家に守られたい | 短命の一族を役目として消費することへ慣れている | 名を呼び、食事、灯り、仕事で一族を生活者として見送る |

### 3.4 対立

- 外的対立: 主、玄冬、灯の減少、世代ごとの時間制限。
- 内的対立: 早く勝つため家族を資源として見る誘惑と、一人の生として残したい思い。
- 関係対立: 汐里は玄冬を愛するほど真相を隠し、玄冬は汐里を愛するほど地上に留まり飢える。
- 最終対立: 千年の責任を「終える・分ける・継ぎ直す」のどれで引き受けるか。

### 3.5 終了時の状態

どの結末でも、プレイヤーは玄冬を単なる星喰い、汐里を単なる犠牲者とは見ない。家譜に積み上げた固有名、形見、討伐、看取りが最後の選択画面へ戻り、「この家がここへ来た」と分かる。

### 4.2 汐里の名はch4で一度だけ初開示する

名前の開示順を一本へ固定し、初回夢だけでなくch4前に到達できる全surfaceを対象にする。

1. M34以後のnew gameは作成時に`flags.m34_narrative_schema=1`を持つ。ch4をqueueした時点の既存`flags.ch4`を完読扱いにしない。
2. ch4の現行冒頭`初代の名は「汐里」`は匿名へ改稿し、最終頁へ新しい唯一の開示beat `擦れていた墨が、夜露のように滲む。——汐里。これが、初代の名だ。`を置く。最終頁へ遷移するstore actionは、表示と同じtransactionで`ch4_completed / reveal_shiori_name`を立て、同文をchronicleへ残す。途中頁では立てない。明示skipも同じactionを通り、同文をrecapとして残す。
3. gsp12は`reveal_shiori_name`後だけ解禁し、初開示でなく余韻へ改稿する。表示文は`「汐里——初代の名だ。口にすると、擦れた家譜の墨が戻るようだ」`とする。
4. 郷の位牌、神々のstage反応、形見の由来、Finaleの固有名表示は同flag後だけ実名を使う。

コメント、内部ID、asset filenameはplayerへ見えないため変更しない。user-visible文字列は上表以外にも`rg -n "汐里" src`を出発点にし、実際のrender経路で分類する。新規開始から`Intro → 家業 → 技 → 星契り → 鍛冶/蔵 → 初回夢 → 夢2/3 → gossip 11 → 灯ノ御山core`までを走り、DOM/accessible name/alt/chronicleに`汐里`0件であることをE2E化する。

#### legacy saveの採用規則

現行`flags.ch4`はsceneをqueueした時点で立つため、完読済みか判別できない。そこで新旧判定は同じstorage keyやsave versionへ依存せず、`flags.m34_narrative_schema`をsentinelにする。**sentinelが無いsaveだけ**、初回migrationで次の保守規則を一度適用する。

```ts
if (flags.m34_narrative_schema !== 1) {
  const legacyReveal = Boolean(
    flags.ch4 ||
    (gossipIndex ?? 0) >= 12 ||
    flags.shioriPhase ||
    flags.endingType !== undefined ||
    flags.cleared
  )
  flags.m34_narrative_schema = 1
  flags.reveal_shiori_name = legacyReveal
  flags.legacy_shiori_recap_pending = legacyReveal
}
```

`flags.ch4`だけが真で実際にはscene未読だったlegacy saveも、再度匿名へ戻さず、初回Homeに`家譜の初代の名は、汐里。千年前に御山へ登った楽士である。`という非modal recapを一度だけ残す。表示後は`legacy_shiori_recap_pending=false`へ保存する。全条件が偽なら匿名のままN0順序へ入る。`ch4 only / gossipIndex 11・12 / shioriPhase / endingType 0・1・2 / cleared / 全偽`のfixtureをv1/v3/v4ごとに固定する。

M34以後のv4回帰fixtureとして、`m34_narrative_schema=1 / flags.ch4=true / ch4_completed=false / reveal_shiori_name=false`を保存→reloadし、実名0、legacy recap 0、ch4が未完了のまま灯の余白から再開できることを固定する。sentinelがあるsaveへlegacy ORを再適用してはならない。

新規saveは`new game → 初回夢 → 夢2/3 → gossip 9〜11 → ch4最終頁 → gsp12`と、ch4を早く満たす経路の両fixtureで、最終の開示beatへ遷移する直前までuser-visible文字列に`汐里`が0件、遷移/skipのtransaction後に1件以上となることをtestする。

## 5. 「随所で見せる」配信設計

物語は長さで3層に分ける。同じsessionで強制表示する全画面sceneは原則1件まで。

- **残響（3〜6秒）**: 操作を止めない一文、環境変化、短い声。閉じる操作不要。
- **発見（8〜20秒）**: 石碑、主の対峙、帰還差分、人物会話。任意で詳細へ。
- **章/夢（45〜120秒）**: 全画面朗読。skip、次へ、後で読む、再読を持つ。

| 場所 | 見せるもの | presentation | trigger | 中断対策 |
|---|---|---|---|---|
| Title | 現在の問いと前回の残響 | `つづきから`下に1行 | save要約 | 本文を増やさない |
| Intro | 匿名の家祖、玄冬、大燈籠、二つの呪い | 11beatを3幕へ束ねる将来案 | new game | 現行skip維持。汐里名はch4まで出さない |
| Home | 「今月の物語」1件 | 今月の要の下、最大2行 | stage/新規解禁 | 詳細は「灯の余白」へ |
| 出立 | この地が千年神話の何を映すか | 地域絵の下に問い1行 | 選択地域 | 攻略情報と別label |
| Dungeon入場 | 土地の顔と主の感情動詞 | 2行、2.5秒、操作停止なし | 初回/再訪差分 | 再訪は短縮 |
| 石碑 | lore fragment＋「これは何を示す？」 | 調べるSheet | monument | 取得後いつでも家譜再読 |
| 稀相予兆 | 過去の言葉が環境へ漏れる | 音/粒子＋5〜12字 | rare seed確定時 | 確率外れ時は出さない |
| 主戦前 | coreを言い直さず、本人の願い | 1〜2台詞＋固有画 | boss contact | 2回目以降短縮可 |
| 戦闘 | 継足/戦闘不能に家族関係の短句 | combat bark、1行 | kin/lowHP/legacy item | 同一戦闘各種1回 |
| 勝利 | 倒したでなく「終われた/残った」 | result上部1行 | boss victory | lootを押し下げない |

| 鍛冶 | 形見が神話の選択を反映 | 前所有者/戦果/最後の一言 | legacy item選択 | stats比較を先に置く |
| 星契り | 神が現在stageへ一言反応 | pact前/後1行 | god affinity + stage | 契約説明と分離 |
| 家系図 | 今代の問い、歴代の選択傾向 | 代nodeの短句 | succession | 結末を誘導しない |

| 再開 | 前回の灯＋未読物語1件 | Home一度 | dormant session | `今は読まない`あり |
| Finale | 家の歩みを3択の前に返す | 固有名3件＋三択 | final boss clear | どれも選択可能 |

### 5.1 「灯の余白」

Homeの記録群へ、未読物語を一か所に集める軽い入口を設ける。新しい大メニューにしない。

- 表示: `未読1 / 今代の問い / 次に手がかりがある場所`。
- 内容: 章、夢、郷の声、地域縁起をsource別でなく時系列に並べる。
- 初期表示は最大5件。全件はfilterで見る。
- `今は読まない`で未読は失われず、主CTAへ割り込まない。
- spoilerは`？？？`の数を並べず、「まだ名のない声が一つ」のように質だけ予告する。

## 6. 家族史を本編へ接続する

### 6.1 一代の問い

継承時に、直前世代の実績から問いを一つだけ作る。性格診断や正解判定ではない。

| 条件例 | 問い例 | 後で返す場所 |
|---|---|---|
| 主を多く鎮めた | 「終わらせることで救えるものは、本当にあるか」 | 主戦、Finale |
| 負傷者を連れ帰った/早期帰還した | 「退くことは、誰かと明日を分けることか」 | Home、夢5 |
| 形見を代重ねした | 「受け継ぐことは、持ち主の役目まで背負うことか」 | 鍛冶、夢7 |
| 地域縁起を多く完成 | 「事情を知った刃は、知らぬ刃より優しいか」 | 図鑑、Finale |
| 多く星契りした | 「血を繋ぐことは、誰の願いなのか」 | Pact、夢7 |

質問は能力補正を与えない。家譜とFinaleで再提示し、プレイヤー自身が意味を決める。

### 6.2 固有名の回収

Finale直前に次のうち最大3件を表示する。

1. 最初に看取った者の名と辞世の短い一節。
2. 最も長く継いだ形見と最初の持ち主。
3. 最も因縁の深い主/宿敵、または最も完成度の高い地域縁起。

対象が存在しない場合は別項目へfallbackする。固定文「数え切れぬ先祖」だけで済ませず、実saveの名を一つ以上読む。

## 7. 三結末への「灯の響き」

内部save互換のため`cut / save / inherit` IDは変えない。ただし`cut`のuser-visible動詞は、既存結末で実際に行う看取りへ合わせて次へ統一する。

- 選択label: `送る — 千年を終わらせる`
- 選択desc: `一族の名を読み上げ、汐里と玄冬を看取り、常夜を終える。`
- 夢8の予告: `二人を送るか、腕を貸すか、それとも——私の座を、継ぐか。`
- 結末本文: 現行どおり刃を納め、家譜の名を読み、二人を看取る。斬撃を追加しない。

表示文、夢8、選択、結果、epilogueの主動詞をbranchごとに比較するtestを追加し、`送る→看取る/夜明け`、`救う→封じ直す/二人の錠前`、`継ぐ→座を受け取る/汐里の解放`が一致しなければ不合格とする。

### 7.1 記録するが、ロックしない

`cut / save / inherit`の三方向に、物語上の行動から小さな`resonance`を記録する。`cut`は内部互換IDで、画面上は`送る`と表す。数値は非表示で、結末条件や報酬倍率にしない。

| 響き | 行動の意味 | 使用例 |
|---|---|---|
| `cut` | 終われないものへ終止符を渡した | 地域縁起のrequiem完成、宿敵決着 |
| `save` | 危険を分け、誰かを独りにしなかった | 負傷者を連れ帰る、親子/兄弟の継足 |
| `inherit` | 役目や記憶を形を変えて次へ渡した | 形見の代重ね、継承、家業の継承 |

Finaleでは「この家は○○を選び続けた」と断定せず、最大方向へ1行だけ添える。

- cut例: `この家は、終われぬ主へ何度も幕を引いてきた。`
- save例: `この家は、傷ついた者を置いて行かない夜を重ねてきた。`
- inherit例: `この家は、名と道具と役目を、形を変えて渡してきた。`

3択の見た目、順序、操作強度は同じにする。「おすすめ」badgeは禁止する。

## 8. Narrative stateと順序保証

### 8.1 最小schema案

```ts
type NarrativeChannel = 'ambient' | 'discovery' | 'scene'
type NarrativeStage = 'one_light' | 'name' | 'duet_hunger' | 'fire_vow' | 'summit'

interface NarrativeBeat {
  id: string
  stage: NarrativeStage
  channel: NarrativeChannel
  trigger: NarrativeTrigger
  text: string
  visualId?: string
  speaker?: string
  once?: boolean
  priority: number
  requires?: string[]
  blocks?: string[]
  recap?: string
}

interface NarrativeProgress {
  stage: NarrativeStage
  seen: string[]
  deferred: string[]
  resonance?: { cut: number; save: number; inherit: number }
}
```

既存save v4を壊さないため、新fieldはoptional。`flags`からmigration時に導出し、旧saveは見た章/夢を再表示しない。

### 8.2 夢の順序修正

`nextDreamEpisode()`は「解禁済みのどれか」ではなく、**最初の未読篇だけを見て、その篇がeligibleなら返す**。

```ts
const next = DREAM_EPISODES.find((ep) => !seen(ep))
return next && eligible(next, data) ? next : null
```

さらに各篇へ`requiresPrevious`を明記し、`yume_maki`の`gen>=5`や`yume_itadaki`の`gen>=8`が前篇を飛ばさないtestを追加する。

初回夢は連作の入口なので、`dreamSeen`を全連作の必須条件にする。初回夢自体は`星骸の谷を制した / 最初の看取りを終えた`のORでeligibleとし、前者へ到達する前に家族が死んでも汐里との初対面だけは先に起こす。初回夢を完読またはskipして`dreamSeen`になった次の月以降に夢渡り・弐を解禁する。同一月に初回夢と弐を連続表示しない。

ただし初回夢では名乗らせず、`dreamSeen`は「家祖らしき楽士に会った」事実だけを表す。実名flag `reveal_shiori_name`はch4最終の開示beatへ遷移するtransaction、またはskip transactionだけで立て、`gsp_12`を含む実名表示は同flagを要求する。M34 new gameは`m34_narrative_schema=1`を初期化し、legacy ORを再適用させない。

### 8.2.1 `NarrativeStage`の単調導出

- `summit`: ch5完読または夢8完読。
- `fire_vow`: ch4完読または夢7完読。
- `duet_hunger`: ch3完読または夢4〜6のいずれか完読。
- `name`: ch2完読または夢2〜3のいずれか完読。
- `one_light`: それ以前。

一度上がったstageは下げない。ただし各beatの`requires`をstageだけで省略せず、`seen` IDで固定する。章で真相を先に知ったplayerにも夢は「別視点の感情回収」として順番どおり残す。

### 8.3 queue規則

- 同一月の全画面sceneは`death > birth/rite > chapter > dream > daily`の最大1件。残りは`deferred`へ送る。
- 夢は看取り直後に「夢が開いた」とだけ知らせ、`今見る / Homeで後で読む`を選べる。
- `deferred`は次月に自動で割り込ませず、Home「灯の余白」へ積む。未読章/夢が7日以上残った時だけ、Homeに非modalの1行通知を一度出す。
- 発見/ambientは全画面scene数へ数えないが、同一行動の直後に最大1件。
- 既読判定はsceneをqueueした時でなく、最後のbeat到達または明示skip時に行う。crash/離脱で未読を失わない。
- skip時は`recap`1〜2文を家譜へ残す。skipしたため重要選択肢を失う設計は禁止。

### 9.1 画像だけで説明しない

- 各篇へ60〜90字の`visualDescription`を持つ。
- 重要な変化は台詞にも一度だけ現れる。例: 「琵琶を地へ置いたのは、これが初めてだった」。
- 画像OFF、通信失敗、読み上げでも同じ因果を理解できる。
- mobileで横長CGを`cover`背景にしない。全篇で最初に16:9全景を必ず見せ、その下へ台詞panelを分離する。

### 9.1.1 画面幅別fit契約

| viewport | 画像frame | fit / position | 必須条件 |
|---|---|---|---|
| 360/390px | 幅100%、`aspect-ratio: 16 / 9`の`figure` | `contain / 50% 50%` | 画像全体、全主焦点、頁/skip/後で読むが同時に到達可能。台詞/controlsを画像へ重ねない |
| 768px portrait | 本文上の16:9 `figure`、最大幅100% | `contain / 50% 50%` | 画像全体を欠かさず表示。余白は夜色＋薄い星粒で処理 |
| 1280px以上 | full-bleed可 | `cover / 50% 50%` | 画像比率がviewportと異なる場合も、篇別の全主焦点が残らなければ16:9 `contain`へfallback |

mobileの余白を埋める場合は、同画像を背面に`cover + blur + opacity`で置いてよいが`aria-hidden`の装飾とし、因果を伝える正本は前面の`contain`画像だけとする。`全景を見る`は拡大用の任意機能であり、初見で欠けた情報を救う必須操作にしない。

篇別の必須焦点は§9のcrop不変条件欄をそのまま用いる。Playwrightで前面`img`のnatural ratio、`object-fit: contain`、frame矩形、controls非交差を取得し、360/390/768px screenshotを人間が全焦点可視として採否する。

### 9.2 既存CGの優先監査

新規画像7点の統合後、次の順だけを再評価する。全CGの一括再生成はしない。

1. `cg_prologue.jpg`: 玄冬・汐里・二つの呪いが絵で読めず、雪景色の家だけに見えるため、Intro再編時に置換候補。
2. `cg_kiro.jpg`: 初回夢coverとしては残すが、人物identityの品質差を確認。
3. `cg_ch1`〜`05`: 各章の主題と画像が一致しない場合だけ差替え。夢CGと同じ事実を描かない。
4. ending6点: 3結末の差が一目で分かるかを確認し、同じ構図の色替えなら優先修正。

## 10. その他の改善点

### 10.1 物語の速度

- 1sessionの強制朗読合計を120秒目安、180秒上限にする。
- 連続sceneが2件以上なら2件目以降へ`後で読む`を出す。
- 初見の主戦前台詞は20秒以内、再戦は5秒以内に短縮する。
- story text中の既知事実を毎回説明せず、「新情報1 / 感情1 / 次の問い1」を上限にする。

### 10.2 郷の変化

- gossip解禁時、話者がいる施設の灯、小道具、立ち位置のうち1つを変える。
- 死者が出た翌月は全郷を暗くせず、その人物と関わった場所一つへ小さな供えを置く。
- 主討伐後は地域由来の品、植物、音が郷の一角へ戻る。Homeの帰還差分から歩いて見に行ける。
- 汐里の名が判明した後、大燈籠前の無名位牌に名前を刻む。大きなcutsceneでなく世界状態の変化にする。

### 10.3 戦闘と物語

- 主の兆しに「攻/術/群」だけでなく、core loreと対応する感情動詞を一つ出す。例: `待ち焦がれる / 守り抜く / 忘れまいとする`。
- 物語文は戦闘判断を邪魔しない。感情動詞はskill effectの代わりではなく、兆しの副label。
- 家族間の継足時、親子/兄弟/形見武器の関係ごとに短句を各戦闘1回だけ出す。
- 戦闘不能者の長台詞は禁止。帰還後または看取りで感情を処理する。

### 10.4 神々

- 180柱へ本編説明を追加しない。物語鍵神8〜12柱だけ、`NarrativeStage`ごとの反応を最大3段持つ。
- 北辰老、大燈台に関わる神、月/天の川/終わりを語る極ツ星を優先する。
- 同じ真相の説明役にせず、「神から見た汐里」「神から見た短命」を短く示す。
- 未読台詞がPactの費用・能力比較を押し下げない。詳細Sheetまたは契り後余韻へ置く。

### 10.5 Accessibility / comfort

- `次へ / 後で読む / skip / 自動送りOFF`を全sceneで同位置にする。
- Space/Enterで次へ、Escapeは進行に使わない。閉じた後は発火元へfocus復帰。
- 台詞の自動送りは初期OFF。導入する場合も速度3段と一時停止を用意する。
- 音が物語上重要な「楽の音」は字幕/視覚波形/短文を併記する。
- 点滅する星、日食、画面flashは演出軽減で静止。色反転を必須情報にしない。
- 読字負荷を下げるため1画面最大3段落、1段落2〜3行。全履歴はSheetで見る。
