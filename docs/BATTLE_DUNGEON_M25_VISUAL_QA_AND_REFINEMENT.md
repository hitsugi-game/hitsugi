# M25 戦闘・ダンジョン実描画検収と空間再設計 — Claude/Fable 5 実装指示

作成日: 2026-07-13  
対象: `main` / M24適用・公開後  
実装担当: Claude/Fable 5  
正典: `docs/GDD_v3.md` §8.10

## 0. この指示の位置づけ

M24はレスポンシブzoom、灯路、灯脈、三段戦闘UI、地域別地面材質まで実装済みである。本指示はM24を作り直すものではなく、実ブラウザで初めて確認できた構図崩れを直し、保留していたPhase Eを品質順に実施するものとする。

優先順位は以下で固定する。

1. P0: PC/モバイルの破綻を直す
2. P0: 戦闘素材を同じ舞台上の表現へ正規化する
3. P1: ダンジョンを「通路の集合」から「記憶に残る場所」へ変える
4. P1: 敵の次行動を判断材料として見せる
5. P2: 既存ルールを壊さない環境反応を足す

粒子、揺れ、発光を増やすだけの対応は禁止する。今回の主題は、情報階層、空間構成、素材の見せ方、操作後の応答である。

## 1. 2026-07-13 実ブラウザ検収結果

### 1.1 検収条件

- ローカルdev版、`main=f11ac8b`
- 1280×720 / 390×844
- 宵の森 地下1層
- 味方1人対敵2体の通常戦
- 出立→加護見送り→タップ移動→接敵→攻撃まで実走
- consoleはlocalhostでGoatCounterを数えない警告のみ。ゲーム由来のerrorなし

### 1.2 ダンジョン — 1280×720

良くなった点:

- 主人公の大きさ、中心光、床とプロップの分離はM24前より改善した。
- タップ入力から接敵まで正常に遷移した。
- HUD、ミニマップ、D-padは機能している。

残る問題:

- 画面下半分近くが意味のない暗部となり、M24受入条件「説明のない純黒面25%未満」を満たしていない。
- 原因は地域backdrop不足ではなく、マップ端でも主人公中心を維持してカメラをクランプしないことが主である。
- 初期画面に進行方向、出口、石碑、敵影のいずれも見えず、何を目指す場所かが分からない。
- 床が同寸の灰色矩形、境界が同型の丸い茂みの反復に見え、森ではなく生成タイルの試作感が勝つ。
- 地名・階層・目的・踏査率は小さく、画面の雰囲気に溶けて判断情報として弱い。
- 右上ミニマップが小さく、現在地と次に行ける空間の関係を読み取りにくい。

### 1.3 ダンジョン — 390×844

- 灯、地名、短期目的、暦、資源、小休止、帰り火、ミニマップが上端で競合する。
- 横幅は確保できているが、縦方向は下半分が暗部になり「歩ける世界」より「闇の空き画面」に見える。
- D-padと隊員札は押せるものの、視線が上端と下端へ分断される。
- safe-areaを考慮した一枚のHUDとして再編する必要がある。

### 1.4 戦闘 — 1280×720

良くなった点:

- 行動順、戦場、コマンドの三段構造は成立した。
- 背景画と主要4コマンドは読み取れる。
- ワンタップ攻撃は快適であり、維持する。

残る問題:

- 味方は縦長の墨絵札、敵は正方形のイラスト、別の敵は小さな暗い札というように、素材の比率・余白・画風が無加工で混在する。
- 大きい敵札が小さい敵札を覆い、HP札と絵の対応が一見で分からない。
- 味方札の白地と赤い落款が戦場背景から浮き、人物ではなく商品画像が置かれたように見える。
- 人物を単純に拡大した結果、人数が増えた時の占有規則がなく、前後関係が偶然の重なりに見える。
- 行動順と戦闘ログは小さすぎ、画面中央の大きな絵との情報密度が釣り合っていない。
- 既存30px踏み込みだけでは陣営間をつなげられず、灯脈が切れた時に攻撃の接触感が弱い。

### 1.5 戦闘 — 390×844

これはM25の最優先修正である。

- 味方1体と敵2体で札同士が大きく重なり、敵名・HP・属性・長印・戦闘ログも同じ帯で衝突する。
- 1体撃破後も残敵札と味方札が交差し、誰が攻撃し誰が残ったかを瞬時に判断できない。
- 行動処理中は下段コマンド領域が大きな空箱になり、故障または入力待ちに見える。
- 背景の魅力はあるが、札の重なりが全面を占めて背景と戦場の奥行きを消している。

## 2. 画面の核となる方針

戦闘と探索の共通テーマは引き続き「灯が闇を読み解く」とする。

- 探索: 闇を広げるのではなく、灯が次の判断地点まで空間構造を読ませる。
- 戦闘: 人物を大きくするのではなく、行動者、対象、結果の三点を灯脈で一つの出来事にする。
- 素材: 画風が異なる画像を無理に切り抜かず、全てを意図された「戦絵札」として同じ枠・比率・接地規則へ入れる。
- 情報: 平常時は静かに、選択時だけ強くする。常時点滅・常時発光は増やさない。

## 3. Phase 1 — ダンジョン画面端・HUD修正(P0)

### 3.1 カメラをマップ境界へクランプする

`src/dungeon/camera.ts` に純粋関数として安全表示領域と境界クランプを追加する。

- PCの安全領域: 上56px、左右0、下0。
- 390px幅: 上96px、左右0、下118pxを操作/UI予約領域として扱う。
- マップが安全領域より大きい軸は、マップ端が安全領域内側へ入らないようcamera targetをclampする。
- マップが安全領域より小さい軸は、主人公ではなくマップ全体を安全領域中央へ置く。
- look-ahead後の座標へclampを適用する。先にclampしてからlook-aheadを足してはならない。
- backdropは端の不足を隠す二次策とし、クランプの代用にしない。
- 画面揺れはclamp後に最大4pxだけ加え、マップ外を大きく露出させない。

追加テスト:

- 四辺と四隅でマップ端が安全領域から離れない。
- 小マップが中央に置かれる。
- PC/390px、zoom最小/最大、look-ahead四方向を網羅する。
- カメラ、灯り穴、screenToTileの既存往復整合を壊さない。

### 3.2 モバイルHUDを一つのレイアウトへする

390px幅では、次の2段だけを上端に置く。

1. 44〜48px: 灯リング / 地名・階層 / 小休止
2. 34〜38px: 短期目的 / 進捗 / ミニマップ拡大

暦・今回の実り・帰り火は小休止sheetへ寄せる。ただし帰り火は2操作以内を維持する。PCでは現行の右側資源表示を残してよい。

- ミニマップ通常表示はPC128px、モバイル104pxを目安とする。
- 拡大時は現在地、発見済み部屋、接続、目的カテゴリを読めること。未発見の答えは見せない。
- D-padは48px以上、端から12px以上、隊員札とは8px以上離す。
- `env(safe-area-inset-*)` を全固定UIへ一度だけ適用し、個別要素で重複加算しない。

### 3.3 タイルの反復感を減らす

地面材質を追加する前に、既存材質の配置規則を直す。

- 歩行主面、壁際、奥壁の明度差を静止画で維持する。
- 同じ茂み/石/茸を3個以上等間隔で並べない。
- 4×4タイル以内で同じ床矩形の色を完全反復させない。
- 主経路は床の向き、落葉、薄い灯痕のいずれかで連続性を持たせる。
- 初期視界には「次の開口」「署名物の気配」「敵の兆し」のうち最低1つを入れる。
- 闇は残すが、何もない純黒と、未踏の森のシルエットを区別する。

## 4. Phase 2 — 戦闘の絵札正規化とモバイル再構成(P0)

### 4.1 素材を共通の「戦絵札」へ入れる

全combatant画像を直接配置せず、共通component `BattleArtFrame` 相当を通す。

- 基本比率は4:5。通常札、宿敵札、主札の3階層だけを持つ。
- `object-fit: contain` 任せにせず、表示専用profileで焦点位置、拡大率、余白、札/透過絵の種別を持つ。
- profileは戦闘表示専用ファイルへ置き、神・敵のゲームデータや戦闘計算へ混ぜない。
- profile未登録は共通defaultへ退避し、raw画像を枠外へ出さない。
- 枠内下部へ暗いグラデーションを置き、白地、透明地、正方形素材の差を同じ舞台照明へ馴染ませる。
- 名前、HP、属性、状態、長印は画像に重ねず、札の直下に固定する。
- 主は大きくしてよいが、通常敵を隠さない専用slotへ置く。

本方針では白い余白を持つ味方画像も欠陥扱いで切り捨てず、「絵札」として意図的に見せる。将来透過立ち絵へ差し替えても同じ枠を使える構造にする。

### 4.2 人数別の配置規則を固定する

Desktop:

- 敵は左42%、味方は右42%、中央16%を灯脈と攻撃結果の空間にする。
- 各陣営はCSS gridを基本とし、通常時の札の相互重なりを12%以下にする。
- 1体、2体、3〜4体を別のslot presetにする。単純な`translateX`の連鎖で並べない。
- 前後列は上下差と小さな列印で示し、z-indexは列とslotから決定論的に算出する。

Mobile 390×844:

- 戦場内を「敵列」と「味方列」の2帯へ分離する。敵札と味方札を同じ奥行きへ重ねない。
- 通常札幅は78〜112px、主は最大148pxを目安とする。
- 生存2体以上では札の重なりを使わず、gridの列数と縮小で収める。
- 戦闘ログは戦場下端の1行専用帯に置き、名前札/HPと交差させない。
- 主要4コマンドはスクロールなし、各48px以上を維持する。
- 札、名札、ログ、下段コマンドの矩形交差を0にする。

### 4.3 攻撃を「元札の長距離移動」で表現しない

元の札は隊列位置を保つ。行動時だけ戦場中央のaction layerを使う。

1. 行動者と対象以外を12〜18%暗くする。
2. 灯脈を行動者の札端から対象の札端まで通す。
3. 近接は行動者の半透明残像、斬線、火花のいずれかを中央へ走らせる。
4. impactで対象札だけを揺らし、HP/UI/コマンド盤は揺らさない。
5. 55〜75msのhit-stop後、灯脈と残像を消す。

灯脈の端点は行動開始時に一度だけ取得する。毎frameの`getBoundingClientRect()`は禁止する。stage shakeを掛ける場合は、札と灯脈を同じtransformコンテナへ入れ、既知LOW「灯脈の切断/rectずれ」を解消する。

### 4.4 入力不能時間を空箱にしない

- 味方行動処理中: 選んだ「攻撃/技名 → 対象名」を表示する。
- 敵行動中: 次項の兆し、または「敵の手番」を表示する。
- 勝敗遷移中: 「戦果を結ぶ」「帰り火へ」等の状態を表示する。
- 入力不能なら主要ボタンはdisabled表示にし、領域自体を空にしない。
- オート状態は常時判別でき、戦闘をまたいで継続する既存挙動を壊さない。

## 5. Phase 3 — 敵の兆し(P1・ロジック別commit)

完全な行動名や数値を見せず、次の判断に必要なカテゴリだけを1手先まで表示する。

- 攻: 単体攻撃
- 群: 全体/複数攻撃
- 術: 状態/属性技
- 守: 防御/強化
- 弱狙: HP割合が低い味方を狙う傾向
- 溜: 次手が強いが今手は準備

規則:

- 敵名札の上に最大2字+小アイコンで表示する。
- 行動決定後に表示し、手番・対象条件が変わって再抽選した時は同じtickで更新する。
- 図鑑未見でもカテゴリは見せる。詳細名称、倍率、正確な対象は土地の観察度や図鑑で段階開示してよい。
- オート戦闘は兆しを参照して新たな最適化をしない。今回はプレイヤー向け情報追加に限定する。
- AI/威力/対象選択を変える変更は、このcommitへ混ぜない。

## 6. Phase 4 — room archetypeと記憶に残る一室(P1)

`src/dungeon/maps.gen.ts` は手編集禁止。`scripts/gen_floor.mjs` と `scripts/gen_all_maps.mjs` を変更し、再生成する。

最低6種を用意する。

1. 門前: 入り口、次の開口、地域の署名を短く見せる
2. 広間: 敵との間合いを選べる
3. 回廊: 視線と追跡の圧を作る
4. 中庭: 署名物/祠/石碑を主役にする
5. 橋・狭路: 一本道だが景観と緊張を持つ
6. 段丘・分岐: 高低差の見立てで二択を作る

生成保証:

- 全通常フロアへ2種以上、全地域の入口または中盤へ「記憶に残る一室」1つ以上。
- 入口から最初の判断地点まで8秒以内を目安にする。
- 行き止まりの100%に宝、石碑、焚火、景観署名のいずれかを置く。
- 同形部屋の3連続を禁止する。
- 見通しの良い直線は12タイルを上限目安とし、超える場合は門、光、障害物で節を作る。
- 主の前室は通常部屋と輪郭、照明、音を変える。

生成検証へ追加するもの:

- archetype数
- 記憶室の有無
- 空の行き止まり数=0
- 同形3連続=0
- 長直線数
- 開始地点から最初の分岐/特殊物までの歩数
- 入口、階段、主、石碑、宝の到達可能性

## 7. Phase 5 — 環境反応(P2・任意)

Phase 1〜4の受入後に行う。新通貨や恒久成長は追加しない。

- 消えた灯籠: 触れると周囲の地形だけ一時的に見える
- 呪い茨: 回り道か、短い演出で通るかを選ぶ。報酬/消費は既存eventへ接続する
- 風鳴り: 敵が近い方向だけ環境音と草の向きで示す
- 隠れ道: 土地の観察度に応じ、壁際の灯痕が見える

各フロア0〜1個に留める。毎回操作を要求してテンポを落とさない。

## 9. 実ブラウザ受入条件

### 9.1 ダンジョン

- 1280×720、1440×900で説明のない暗部が画面の15%以下。
- 390×844で説明のない暗部が20%以下。
- 四辺・四隅でマップ外露出が増えず、主人公と灯り穴がずれない。
- 上端HUDの要素交差0、D-pad/隊員札/帰り火/safe-areaの交差0。
- 初期画面3秒以内に、歩ける方向と短期目的を説明できる。
- ミニマップ拡大で発見済み部屋のつながりを説明できる。
- 到達可能タップは100ms以内に目的印、250ms以内に移動開始。
- 到達不能タップは割れ印を返す。

### 9.2 戦闘

- 1280×720と390×844で、味方1/4、敵1/2/4、主1の各画面を撮る。
- 平常時の生存札の相互重なりは12%以下。名札、HP、ログ、コマンドの矩形交差0。
- 390×844の味方1対敵2で、全員の名前・HP・現在手番を500ms以内に特定できる。
- 画像の元比率に関わらず、全員が同じ戦絵札の光、枠、接地へ収まる。
- 行動処理中に下段が空箱にならない。
- 攻撃時、行動者→対象→結果が1本の灯脈/残像で追える。
- 被弾でHUDとコマンド盤が揺れず、灯脈端点が札から外れない。
- 主要4コマンドは390×844でスクロールなし、44×44px以上。推奨48px以上。
- reduced-motionでも対象、結果、兆しを失わない。

### 9.3 性能・機械検証

- ダンジョン10秒平均: PC 55fps以上、標準的モバイル30fps以上。
- 戦闘10秒平均: PC 55fps以上、標準的モバイル30fps以上。
- fps計測用表示は開発限定とし、本番へ常設しない。
- 新規全画面filterを追加しない。action layerのDOM計測は行動開始時1回まで。
- `npm run build`
- `npm run lint`
- 全vitest
- `node scripts/validate_data.mjs`
- `git diff --check`

## 11. Claudeへ渡す最初の実装指示

まずPhase 1とPhase 2だけを実装する。Phase 3以降へ進む前に、次の4画面を同じセーブで撮影する。

1. 1280×720 宵の森 地下1層の開始地点
2. 390×844 同地点
3. 1280×720 味方1対敵2
4. 390×844 同戦闘

この4枚で暗部率、札重なり、名札/ログ/コマンド交差が合格しなければ、素材追加やroom archetypeへ進まない。
